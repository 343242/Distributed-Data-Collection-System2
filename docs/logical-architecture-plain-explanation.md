# 分布式数据采集系统为什么这样设计（大白话版）

本文对应 [logical-architecture.md](/home/instant/projects/Distributed-Data-Collection-System2/docs/logical-architecture.md) **v0.9** 版本。如果正式版迭代到 v1.0 及以后，请以正式版为准；本文件中的解释可能需要同步更新。

```text
这套系统为什么要这样设计，而不是更简单一点？
```

---

## 1. 先说人话：这套系统到底在解决什么问题

这套系统不是普通的“把数据传上来”。

它面对的是客户现场的数据采集场景，现实情况通常是：

- 数据源很多，点位很多，采集频率还高。
- 客户现场网络不一定稳定。
- 中心服务也可能升级、抖动、限流甚至短时不可用。
- 一旦现场断网，数据不能直接丢。
- 一旦中心恢复，之前积压的数据还要补回来。
- 同时还得知道是谁在采、采得对不对、出了问题能不能查。

所以这套系统的核心目标不是“链路能跑通”，而是：

- 平时稳定采。
- 出故障尽量不丢。
- 恢复后能继续补。
- 出问题知道卡在哪。
- 后面规模变大时还能继续演进。

这就是为什么它不能只做成一个“Connector 直接把数据写数据库”的简单程序。

---

## 2. 为什么先引入 Customer

因为不引入客户这个对象，很多问题会一直说不清。

比如：

- 这个 Agent 属于谁？
- 这个任务归谁？
- 这批数据是谁的？
- 哪些人能看、能改、能停？
- 某个客户把资源打满了，会不会影响别的客户？

所以系统第一层边界不是 Agent，也不是任务，而是 Customer。

说白了，Customer 的作用就是一句话：

```text
先把“谁家的东西”说清楚，后面权限、隔离、审计、配额才有根。
```

---

## 3. 为什么要有 Agent，而不是让 Connector 直接连中心

因为 Connector 只是“干活的采集程序”，不适合承担现场治理职责。

如果让 Connector 直接面对中心，它就得自己处理很多本来不该它处理的事：

- 怎么注册
- 怎么拿配置
- 怎么升级
- 怎么重启
- 怎么缓冲
- 怎么重传
- 怎么限流
- 怎么上传日志

这样每种 Connector 都要重复造一遍轮子，而且不同类型很容易做得参差不齐。

所以这里把 Agent 设计成现场的“总代表”：

- Gateway 只管 Agent，不直接管具体采集进程。
- Connector 只负责采数据。
- 现场的进程管理、缓冲、上传、恢复、日志都由 Agent 统一承担。

这样做的好处很直接：

- Connector 更容易开发。
- 不同 Connector 的行为更统一。
- 现场治理能力集中在 Agent，后续更好维护。

---

## 4. 为什么数据一定要先到 LocalBuffer

因为网络和中心服务都不值得完全信任。

如果 Connector 采到一条数据后直接往中心发，那么中间任何一个环节出问题，数据就可能丢：

- 刚发出去网络断了
- 中心收到了但 ACK 丢了
- 中心过载暂时写不进去
- Agent 或 Connector 正好重启

所以第一版要求所有数据先进入 Agent 本地可靠缓冲，也就是 `LocalBuffer`。

这背后的思路很朴素：

```text
先把数据接住，再谈上传。
```

只有 Agent 明确把数据写进本地缓冲后，Connector 才能认为“这批数据已经安全了”。

这样即使中心短时不可用，现场也还能继续采；等网络恢复，再把积压数据慢慢补上去。

---

## 5. 为什么数据链路还要经过 Gateway，而不是直接进数据存储

这是很多人第一眼都会问的问题，因为表面看起来多了一跳。

大白话讲，Gateway Ingest Plane 的价值不是“帮你转发一下数据”，而是“把数据入口统一管起来”。

如果不经过 Gateway，很多关键问题就没地方统一收口：

- 这批数据是不是这个客户的
- 这个 Agent 还有没有资格上传
- 这个任务是不是已经被停了
- 这批数据是不是重复上传
- ACK 到底什么时候能算成功
- 协议版本不兼容怎么办
- 某个客户狂发数据时怎么限流

如果没有 Gateway，这些能力就只能散落到 Agent、数据存储前置层或者下游系统里。结果就是：

- 边界不清楚
- 逻辑到处重复
- 问题排查更麻烦
- 后续换存储、扩链路、做审计会很痛苦

所以经过 Gateway 的本质，是统一数据入口治理，而不是单纯多一层网络跳转。

---

## 6. 为什么 Gateway 内部还要拆成控制面和数据面

因为这两类事情本质上不是同一类工作。

控制面关心的是：

- Agent 注册
- 任务启停
- 配置版本
- 程序包下发
- 审计和告警

数据面关心的是：

- 大流量接收批次
- 校验归属
- 幂等判断
- 写入数据存储
- 返回 ACK

前者是低频但很关键，后者是高频而且吃吞吐。

如果两者混在一起，会出现一个很典型的问题：

```text
数据流量一高，连任务启停、禁用 Agent、查看状态这些管理动作也可能一起被拖慢。
```

所以第一版即使物理上可以先部署在一起，逻辑上也必须拆开。这样后面数据面扩容，不会把控制面一起拖进复杂度泥潭。

---

## 7. 为什么不用“一个点位一个任务”

因为那样在工业过程点场景下会直接炸。

一个采集系统可能面对上万个点位。如果每个点位都做成一个任务，那系统里会同时膨胀出海量：

- 任务对象
- 任务状态
- 日志
- Checkpoint
- 监控指标
- 审计记录

最终结果不是“更精细”，而是“整个控制面失控”。

所以第一版把最小控制粒度定成 `CollectorTask`，一个任务对应：

```text
一个数据源引用或设备组 + 一套采集配置
```

点位仍然存在，但点位属于任务内部采集范围，不是第一版的最小启停单元。

这样做的本质是：

- 在控制粒度和系统复杂度之间取一个能落地的平衡点。

---

## 8. 为什么第一版固定为 1 个任务对应 1 个实例

因为第一版要先把可靠性和状态收敛做稳，不是先把并行调度做花。

如果一个任务同时拆给多个实例或者多个 Agent 去跑，马上会带来很多复杂问题：

- 谁是当前 owner
- 谁来推进 Checkpoint
- 多个实例的数据边界怎么划
- 一个成功一个失败时怎么算
- 迁移过程中是不是会重复采

这些问题不是不能做，而是第一版没必要一上来就做。

所以第一版先定成：

```text
1 CollectorTask : 1 ConnectorInstance
```

这样状态、重试、审计、运维、Checkpoint 都更容易讲清楚。

不是以后永远不扩展，而是先把最小可靠闭环做出来。

---

## 9. 为什么选择“至少一次”，而不是“绝对不重复”

因为在这种场景里，“绝对不重复”往往比“允许少量重复但尽量不丢”贵很多，而且实现风险更高。

只要系统存在这些现实情况：

- 网络闪断
- ACK 丢失
- Agent 重启
- Gateway 写成功但响应没回来

那就会出现一个经典问题：

```text
这批数据到底算成功了，还是没成功？
```

如果结果不确定，系统要想不丢数据，最安全的做法就是重传。  
一重传，就必须接受“可能重复”这个现实。

所以第一版明确采用“至少一次”语义：

- 允许重复
- 不能静默丢
- 靠幂等机制把重复控制住

这不是退而求其次，而是更符合工业采集第一版的工程取舍。

---

## 10. 为什么要有 messageId、幂等和 ACK 闭环

因为“允许重传”这件事，必须建立在“重复能识别”之上。

系统里最怕的不是重复一次，而是重复以后分不清：

- 这是老数据重传
- 还是新数据
- 还是同一个编号但内容变了

所以设计里要求：

- Agent 在写入 LocalBuffer 时生成稳定 `messageId`
- 同一条数据重传时 `messageId` 不变
- Gateway 用幂等键识别“这条我以前见过没有”

这样一来，哪怕 ACK 丢了，Agent 重传时 Gateway 也能判断：

- 之前确实成功过，那这次直接按成功处理
- 如果编号一样但内容不一样，那是严重错误，必须报错

这就是为什么文档里反复强调：

```text
数据写入、幂等记录、ACK 语义必须形成一致性闭环。
```

因为只要这里松掉，系统不是重复失控，就是静默丢数。

---

## 11. 为什么要等 Gateway SUCCESS 后才能推进 Checkpoint

因为 Checkpoint 的本质不是“我采到了”，而是“我可以放心从这里往后继续了”。

如果 Gateway 还没明确返回成功，Agent 就抢先推进 Checkpoint，会有一个很严重的后果：

- 源端游标已经往前走了
- 但中心其实没真正落下来

这样一旦故障恢复，系统会以为这部分数据已经处理完，最终造成真正的数据丢失。

所以规则必须很死：

- Connector 收到 `append accepted`，只代表进入了本地可靠缓冲
- Agent 收到 Gateway `SUCCESS ACK`，才代表中心侧闭环完成
- 只有这时候，任务级 Checkpoint 才能推进

说白了就是：

```text
Checkpoint 不是“采集成功”的确认，而是“中心确认收到了”的确认。
```

---

## 12. 为什么部分成功不做细粒度断点续传，而是整批重传

因为第一版最怕把简单问题做复杂，还做不稳。

如果支持批次内部分成功、部分失败，再从中间断点续传，就要额外解决很多事情：

- 批次内哪些记录成功了
- 哪些没成功
- Checkpoint 候选值怎么归并
- 已成功部分怎么避免和未成功部分冲突
- 崩溃恢复后怎么还原中间状态

这一套做下来，复杂度会显著上涨。

所以第一版选了更保守的策略：

- 结果不确定，就整批重传
- Gateway 用幂等机制识别重复

这套方案的代价是可能多传一些，但换来的是语义清楚、实现稳定、排障更直接。

第一版更需要这个。

---

## 13. 为什么 LocalBuffer 不直接上 Kafka，或者把每条数据都塞进 RocksDB

因为 Agent 是部署在客户现场的，不是公司机房里的重型中间件节点。

如果每个 Agent 都上 Kafka，理论上功能很强，但现实代价也很高：

- 部署更重
- 运维更复杂
- 资源占用更大
- 故障排查门槛更高

如果把每条样本主体数据都写进 RocksDB / Pebble 这种 KV 引擎，也会面临：

- compaction 带来的资源抖动
- 高频写场景下写放大明显
- 本地维护复杂度提升

所以第一版选的是更贴近场景的方案：

```text
顺序追加二进制段文件 + WAL + 回放游标
```

原因很简单：

- 这种场景本来就是高频追加写
- 顺序写磁盘更稳
- 崩溃恢复逻辑更可控
- 本地工具链也更容易做

它不一定是最炫的方案，但很符合“高频写、尽量稳、便于维护”的目标。

---

## 14. 为什么还要设计背压、限流和削峰填谷

因为系统不是只考虑“正常时候能跑多快”，还要考虑“顶不住的时候别一起崩”。

比如这些情况：

- 某个 Agent 现场积压一个月数据，恢复后猛冲中心
- 多个客户同一时间集中补传
- 中心存储暂时变慢
- 某类 Connector 采样量突然抬高

如果没有背压和限流，常见结果就是：

- Agent 本地磁盘打满
- Gateway 被打爆
- 某个大客户影响其他客户
- 控制面和数据面一起抖

所以系统要能做几件事：

- Agent 自己感知积压
- Gateway 对入口限流
- 高峰时放慢上传节奏
- 必要时暂停新采集，先保住已采到的数据

这套设计说白了就是：

```text
系统顶不住的时候，要学会慢下来，而不是硬撑到一起死。
```

---

## 15. 为什么日志、监控、告警要放进架构里，而不是以后再补

因为这种系统最怕“出了问题，谁都说不清”。

现场节点在客户侧，很多时候你不可能直接上机器手工看。  
如果架构阶段不把监控、日志、告警纳进去，后面很容易变成：

- 数据没上来，但不知道卡在采集、缓冲、上传还是中心写入
- 任务停了，但不知道是配置错、权限错、程序崩了还是磁盘满了
- 积压很高，但不知道是网络问题还是限流问题

所以文档把这些直接作为架构能力，而不是实现细节：

- Agent 要上报状态和摘要
- Gateway 要汇总监控视图
- 日志要支持按需拉取片段
- 高风险操作要留审计

本质上是在避免系统变成“黑盒”。

---

## 16. 为什么安全、版本和兼容性要提前卡住

因为这是一个分布式、跨客户、跨现场、长期运行的系统。

如果不提前定义：

- 谁能注册 Agent
- 谁能停任务
- 程序包怎么签名校验
- 协议怎么升级
- 旧 Agent 和新 Gateway 能不能一起跑

后面会很快遇到这些麻烦：

- 客户现场随便接入一个假 Agent
- 程序包被错误替换
- 中心升级后老 Agent 全挂
- 同一系统里不同版本行为不一致

所以文档里把这些看成架构边界，而不是“上线前再加”。

原因很现实：

```text
这类问题一旦带着模糊边界进入实现，后面返工会很贵。
```

---

## 17. 为什么第一版故意不做一些“看起来更高级”的能力

因为第一版最重要的不是功能列表好看，而是基本盘稳。

所以文档明确没做这些事情：

- 不做严格 exactly-once
- 不做批次内断点续传
- 不做自动动态分发
- 不做一个任务多 Agent 并发执行
- 不做自动任务迁移

不是因为这些能力没价值，而是因为它们都会明显拉高：

- 状态复杂度
- 一致性复杂度
- 运维复杂度
- 失败恢复复杂度

第一版先把下面这条线走通更重要：

```text
采得到 -> 现场接得住 -> 中心收得下 -> 出故障能恢复 -> 运维看得清
```

只要这条线稳了，后面的高级能力才有资格往上叠。

---

## 18. 用一句话概括整套设计

如果只用一句话总结这份架构文档，可以这样说：

```text
这套设计不是为了让链路“最短”，而是为了让系统在现场复杂环境下“尽量不丢、出了问题能恢复、规模变大还能管得住”。
```

再展开一点，就是四个核心判断：

- 用 Customer 把系统边界先立住。
- 用 Agent 把现场治理统一起来。
- 用 LocalBuffer 把数据先接住。
- 用 Gateway Ingest 把中心入口统一管住。

这四个点，就是整份逻辑架构文档反复做取舍时的主线。
