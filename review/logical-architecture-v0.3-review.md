## 总体评价

v0.3 相比 v0.1 进步显著，已从"架构骨架"演进为具有相当完整性的逻辑架构方案。v0.1 评审指出的 23 项具体问题中，12 项已完整解决，8 项部分解决，3 项仍未触及。

v0.3 新增的工业高频采集容量基线、二进制流本地缓存模型和削峰填谷策略质量较高，体现了对场景约束的深入理解。

**文档距离可进入详细设计的差距**主要集中在两个维度：

1. **部分设计仍停留在"规定原则"而非"约束实现"** — 如幂等索引性能、Gateway HA 细节、GatewayMetadataStore 一致性模型
2. **运维保障领域仍整体缺失** — 测试策略、灾备恢复、可观测性集成等

但核心数据链路的正确性基础（至少一次语义、Checkpoint 原子性、幂等写入一致性）已在逻辑层面具备可推导的完备性。

### 部分解决（8 项）

#### [H1] Gateway HA 策略 — 部分解决

**已解决**: v0.3 行 125-133 给出 HA 原则：无状态设计、状态外部化、固定域名/负载均衡入口、控制面与数据面隔离保护。

**仍缺失**:
- GatewayMetadataStore 和 IdempotencyStore 的复制策略（同步/异步、RPO/RTO）
- Gateway 实例故障检测和切换对 Agent 的透明度要求
- 多实例部署下幂等写入的跨实例协调机制

**评价**: 已解决"HA 方向"问题，未解决"HA 正确性约束"。对第一版 10 Agent 基线可接受，进入详细设计时需补充。

#### [M1] Agent-Gateway 通信协议 — 部分解决

**已解决**: 第 23 节定义了协议版本字段和兼容原则。

**仍缺失**:
- 心跳间隔和超时的逻辑上限（不是具体值，而是"心跳失败多久触发状态变更"的逻辑规则）
- 控制通道和数据通道是否必须物理分离的约束

#### [M2] 规模目标和性能基线 — 部分解决

**已解决**: 第 19 节补充了详尽的数据面容量基线和性能目标，是 v0.3 最有价值的补充。

**仍缺失**:
- 端到端延迟 SLI 的 P50/P95/P99 目标值（行 793 已列为核心 SLI 但未给数值）
- Gateway 控制面性能基线
- 性能基线的验证方法和验收标准

#### [M5] 幂等索引存储与性能 — 部分解决

**已解决**: 第 13 节强化了幂等写入一致性规则和异常处理。

**仍缺失**:
- 幂等索引的保留策略（TTL？按 Checkpoint 清理？永久保留？）
- 150000 samples/s 吞吐下的存储增长估算
- 热路径查找延迟要求
- 幂等索引与数据写入不在同一事务时的等价方案仅提到需求未给出约束

**风险**: 这是系统瓶颈的最大风险点。

#### [M8] 安全模型规定性 — 部分解决

**已解决**: 从叙述性升级为结构化规定（Bootstrap Token/Agent Credential 两阶段、TLS 强制、进程级隔离、敏感配置加密）。

**仍缺失**:

- LocalBuffer 静态加密是否要求

#### [M9] 多租户隔离 — 部分解决

**已解决**: 第 18 节定义了 customerId/agentId/taskId 三级限流。

**仍缺失**:

- 令牌泄露后的爆炸半径控制

#### [L6] Agent 注册流程 — 部分解决

**已解决**: 定义了注册策略和两阶段凭证。

**仍缺失**: 注册信息模型、Agent 替换后旧注册信息处理、Bootstrap Token 生成和分发流程。

## 二、v0.3 新增内容评审

### 1. 工业过程点高频采集容量基线（第 19 节）

**评价: 高质量，是 v0.3 最有价值的补充。**

优点:

- 容量基线面向具体场景（工业过程点），提供了从 10 Agent / 10000 Point / 1s 周期到 Gateway 设计余量的完整推导链。
- 区分了日常平均（4000 samples/s）和峰值（10000 samples/s），Gateway 设计余量（150000 samples/s）为 10 倍日常吞吐。
- 本地容量规划以 payload 50B 为参考估算了 30 天保留的磁盘需求，从日常平均 518GB 到高保障部署 2.5-4TB SSD/NVMe 的分级建议实用。

风险:

- **50B payload 假设需验证**。如果 payload 包含设备元数据、质量码、扩展字段，实际可能达 200B-500B，30 天保留从 518GB 膨胀到 2-5TB。文档未标注此为必须验证的关键假设。
- **Gateway 吞吐基线缺少延迟约束**。100000 samples/s 峰值吞吐下，Ingest 队列积压可能导致延迟膨胀，但端到端延迟 SLI 目标值未给出。
- **单 Ingest 实例 50000 samples/s 目标缺少依据**。未说明基于什么存储后端和硬件假设，是否已考虑幂等索引查找开销。

### 2. 本地二进制流缓存模型（第 12 节）

**评价: 设计质量高，技术选型合理。**

优点:

- 二进制流 Segment + WAL + ReplayCursor 架构选型正确。明确否定了 Kafka（部署运维重）和 RocksDB/Pebble（compaction 风险），理由充分。
- 一致性规则定义精确：WAL fsync 后可恢复、Segment fsync 后可回放、不确认半个 block、WAL checkpoint 触发条件明确。
- cachectl 运维工具集覆盖 stats、inspect、verify、repair、benchmark，体现运维友好性。

风险:

- **Block 默认 1MB 选择缺少依据分析**。在高频采集场景下（10000 samples/s * 50B = 500KB/s），约 2s 填满一个 block，fsync 频率与 block 填满频率的关系需分析。
- **Segment 默认 256MB 清理粒度偏大**。按任务清理时可能造成空间浪费，特别是任务数据量不均匀的场景。
- **ReplayCursor 持久化位置未明确**。如果 cursor 与 WAL checkpoint 绑定，精度最高为 WAL checkpoint 粒度（64MB 或 5s），可能导致重启后重复回放一小段数据。

### 3. Gateway 接入吞吐目标与削峰填谷策略（第 19 节后半段）

**评价: 设计思路正确，是本文档最具原创性的贡献。**

优点:

- UploadRateController 的输入/输出模型完整，涵盖采集速率、上传速率、缓冲水位、Gateway 反馈、本地资源等多维信号。
- 正常/catch-up/过载降速/恢复探测四档调速策略实用且可配置。
- catchUpEta 计算提供可量化的恢复进度指标，6h/24h 两档告警阈值合理。

风险:

- **catchUpEta 公式在持续 catch-up 速率仍低于采集速率时无解**。分母 `max(uploadRate - ingestRate, epsilon)` 趋向 epsilon 时 catchUpEta 趋向无穷。文档未定义"永远追不上"时的升级策略（降采样？主动降采集频率？等待磁盘满后背压停止？）。
- **恢复探测的振荡风险**。多 Agent 同时探测可能导致 Gateway 反复过载 -> 全部降速 -> 同时恢复 -> 再次过载。缺少探测随机化或交错约束。
- **多 Agent 并发 catch-up 缺少全局协调**。10 个 Agent 各 30000 samples/s 峰值配额 = 300000 samples/s，是 Gateway 设计余量的 2 倍。未定义多 Agent 并发 catch-up 时的协调机制。

## 三、新发现的问题

### 高优先级

#### [N1] 幂等索引的存储增长和性能可行性未建模（延续 M5）

- **位置**: 第 13 节
- **风险**: Gateway 设计余量 150000 samples/s，幂等索引每天新增约 130 亿条记录。即使每条仅存储幂等键（约 60-80B），每天增长约 780GB-1TB。
- **缺失内容**:
  - 幂等索引的保留策略和清理机制
  - 热路径查找的延迟要求（每条记录写入前必须查幂等，可能是同步操作）
  - 索引是否随 Checkpoint 推进清理
  - 150000 samples/s 下的索引存储增长估算
- **建议**: 在详细设计前完成幂等方案的高层选型（布隆过滤器 + TTL 索引？数据表内唯一约束？时间分区索引？），并在逻辑架构中记录选型约束

#### [N2] 多 Agent 并发 catch-up 缺少全局协调机制

- **位置**: 第 19 节
- **风险**: 10 个 Agent 同时进入 catch-up 模式，每个 Agent 峰值 30000 samples/s，总负载 300000 samples/s，远超 Gateway 设计余量 150000 samples/s
- **建议**: 增加约束"Gateway 应具备全局 catch-up 调度能力，在多个 Agent 同时积压时按优先级或配额分配恢复带宽"，或明确列为第一版已知运维约束

#### [N3] writeSeq 与 localSequence 的关系未明确

- **位置**: 第 12 节 (429) vs 第 13 节 (484)
- **现状**: `writeSeq` 由本地存储系统生成、单进程内单调递增（行 429-430），用于 WAL 和回放顺序；`localSequence` 是 messageId 的组成部分（行 484），用于幂等去重。两者是否同一值？
- **风险**: 如果不同，存在映射问题；如果相同，messageId 生成与存储层 writeSeq 耦合，影响存储层灵活性
- **建议**: 在逻辑架构中明确两者关系

### 中优先级

#### [N4] catchUpEta "永远追不上"场景缺少升级策略

- **位置**: 第 19 节 (766-770)
- **风险**: 持续 catch-up 速率仍低于采集速率时，系统需要明确升级路径
- **建议**: 定义从 catch-up 告警升级为背压动作的规则（如降低采集频率、暂停低优先级任务）

#### [N5] GatewayMetadataStore 的一致性模型未定义

- **位置**: 第 4 节 (36, 128)
- **缺失**: assignmentVersion 和 desiredStateVersion 的并发更新需要什么级别的一致性？
- **建议**: 至少定义"读己之写"一致性 — Agent 心跳中携带的 observed version，Gateway 读取时必须看到不早于该 Agent 最近一次写入的版本

#### [N6] 50B payload 假设未标注为需验证的关键假设

- **位置**: 第 19 节 (722)
- **风险**: 此假设直接影响所有容量规划数字的可靠性
- **建议**: 标注为"必须在详细设计中按实际 ConnectorType 输出重新验证的关键假设"

### 低优先级

#### [N7] 恢复探测可能引发多 Agent 振荡

- **位置**: 第 19 节 (760)
- **建议**: 增加恢复探测应引入随机化抖动的约束

#### [N8] Segment 清理粒度可能偏大

- **位置**: 第 12 节 (434)
- **建议**: 评估是否支持 Segment 内部分清理或更小 Segment 选项

#### [N9] v0.1 评审指出的部分完全缺失领域仍未触及

v0.1 列出 10 个完全缺失领域中，v0.3 部分覆盖了 API 版本控制（第 23 节）、限速配额（第 18 节）、数据 Schema 演进（行 347）、容量规划指南（第 19 节）。以下仍完全缺失：

| 领域 | 生产影响 |
|------|----------|
| 测试策略 | 混沌工程、故障注入、可靠性验证方法 |
| 灾备恢复 | Gateway 数据中心丢失时的恢复计划 |
| 可观测性集成 | 指标导出格式（Prometheus/OpenTelemetry） |
| 数据保留与合规 | 采集数据保留期限、数据主权 |
| Agent 自更新机制 | Agent 二进制文件补丁管理 |
| 配置漂移检测 | Agent 本地配置被手动修改的检测 |

## 四、文档优点

1. **v0.1 问题的解决率高** — 23 项中 12 项完整解决，说明迭代过程有效

5. **状态收敛协议完整** — 四层状态模型 + observed version + 乱序拒绝 + 超时告警
