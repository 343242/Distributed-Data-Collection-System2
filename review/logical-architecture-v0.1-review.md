# 逻辑架构文档 v0.1 评审记录

- **评审对象**: `docs/logical-architecture.md` (commit 305d648, tag v0.1)
- **评审日期**: 2026-04-29
- **评审角色**: Architect Agent (Opus)
- **文档版本**: v0.1

---

## 总体评价

文档对 v1 分布式数据采集系统的核心对象模型、数据链路和可靠性策略设计清晰，是一份合格的高层架构骨架。核心内循环（Agent → LocalBuffer → Gateway → Storage）阐述得当。

**根本性问题**：文档描述了架构"是什么"，但对分布式系统边界的交互规范不足——网络分区、崩溃恢复原子性、多租户安全、Gateway HA 均处于规范不足状态，可能导致不同工程师实现分歧。

**三大短板**：

1. **运维维度不足** — 安全、网络分区、背压、资源隔离
2. **一致性语义不够精确** — Checkpoint 原子性、messageId 唯一性、配置冲突解决仅停留在原则声明
3. **扩展性停留在口号** — Gateway 高可用仅一句话，未给出拆分或扩展方向

## 发现问题清单

### 高优先级

#### [H1] Gateway 高可用策略缺失

- **位置**: 第 23 行、第 59-79 行
- **现状**: Gateway 被描述为"统一中心服务"，单点入口，HA 仅在第 539 行一句话提及
- **风险**: Gateway 是中心关键节点，单点故障将导致全系统不可用
- **缺失内容**:
  - Gateway 是有状态还是无状态？
  - 元数据存储如何复制？
  - 故障切换期间 Agent 行为如何？
  - 控制面和数据面是否可以独立扩展？
- **建议**: 在后续版本中补充 Gateway 内部架构拆分方案和 HA 策略

#### [H2] Checkpoint 持久化原子性未定义

- **位置**: 第 219-236 行
- **现状**: 第 233 行规定"只有 Gateway 确认持久化成功后，Agent 才能推进 Checkpoint"，但未说明 Checkpoint 如何原子存储
- **风险**: 如果 Agent 在更新索引之后、Checkpoint 持久化之前崩溃，Checkpoint 可能指向已清理数据或比实际更旧，导致跳过源记录或无限重复
- **缺失内容**:
  - Checkpoint 是否与批次 ACK 持久化到同一个 LocalBuffer？还是单独文件？
  - Checkpoint 更新和批次状态清理是否作为单个事务？
  - 崩溃恢复时如何保证 Checkpoint 一致性？
- **建议**: 明确定义 Checkpoint 持久化的原子性模型，这是"至少一次"语义的正确性基础

#### [H3] 网络分区策略完全缺失

- **位置**: 第 268-273 行（仅覆盖 Agent 干净离线场景）
- **缺失场景**:
  - **脑裂**: 网络分区阻止心跳，但 Agent 仍在运行并产出数据，Gateway 标记 OFFLINE，分区恢复后 Agent 积压大量数据如何协调？
  - **单向分区**: Agent 能收到 Gateway 命令但上传失败，或能上传但心跳失败
  - **网络抖动**: 间歇性连接导致 ACK 丢失，Agent 重传，Gateway 重复写入
- **建议**: 补充网络分区行为章节，覆盖超时、重试策略和状态协调

#### [H4] Connector 程序包签名验证仅为"建议"

- **位置**: 第 391 行
- **现状**: "建议支持签名校验"，非强制
- **风险**: Connector 在客户网络上作为任意代码执行，程序包被篡改是全系统最高影响的安全事件。哈希校验仅防意外损坏，不防恶意篡改
- **建议**: v1 必须将签名验证改为强制要求

#### [H5] Gateway 元数据存储未建模

- **位置**: 第 25 行仅提到"数据存储"用于采集数据
- **现状**: Gateway 自身的操作数据库（任务定义、Agent 注册、Checkpoint 历史、程序包注册表、审计日志）从未作为独立对象建模
- **风险**: 实现时缺乏统一指导，各模块自行选择存储方案导致碎片化
- **建议**: 将 Gateway 元数据存储作为明确组件纳入对象模型

### 中优先级

#### [M1] Agent-Gateway 通信协议未指定

- **位置**: 第 85-86 行（"注册并维持心跳"）、第 92-93 行（"上传批量数据"）
- **缺失内容**:
  - 协议选型：gRPC？REST？WebSocket？TCP？
  - 心跳间隔、超时、重连退避策略
  - 数据上传通道与控制通道是否复用
- **建议**: 明确协议选型（推荐 gRPC，支持双向流），定义认证机制和版本协商

#### [M2] 规模目标和性能基线缺失

- **位置**: 第 543 行（已承认但未补充）
- **缺失内容**:
  - 预期支撑多少 Agent？
  - 每个 Agent 多少 Connector？
  - 数据吞吐量级（条/秒）？
  - 延迟要求？
  - LocalBuffer 容量目标？
- **建议**: 定义 v1 性能基线，决定单 Gateway 是否可行

#### [M3] RETRYABLE_FAILURE 重试策略未定义

- **位置**: 第 215 行
- **缺失内容**:
  - 退避策略（指数退避？固定间隔？）
  - 最大重试次数
  - 最终升级为 FINAL_FAILURE 的条件
  - 重传批次与正常批次对上传带宽的争用
- **建议**: 补充重试策略规范，包括退避参数、最大重试次数和升级路径

#### [M4] LocalBuffer 隔离策略未定

- **位置**: 第 173-188 行
- **缺失内容**:
  - 任务间共享一个 RocksDB 实例还是每任务独立实例？
  - 磁盘满时按任务逐出的优先级逻辑
  - 共享存储的 compaction 对 CPU/IO 的影响
  - 单 Agent 磁盘空间规划
- **风险**: 单个嵌入式 KV 存储损坏影响所有任务
- **建议**: 明确隔离策略并记录权衡（独立实例增资源开销但限制爆炸半径）

#### [M5] 幂等索引的存储与性能未建模

- **位置**: 第 206 行
- **现状**: `customerId + taskId + messageId` 做记录级幂等，需存储并查询每条接收记录的键
- **风险**: 高吞吐（百万条/秒）下去重索引成为瓶颈
- **缺失内容**:
  - 去重索引存储在哪里？
  - 索引保留策略（TTL？）
  - 热路径上的查找延迟
  - 索引自身的存储增长
- **建议**: 对去重索引的存储、保留和性能特征进行建模

#### [M6] 整批重传的故障放大效应

- **位置**: 第 170 行
- **现状**: "部分成功时第一版不做批次内断点续传，采用整批重传"
- **风险**: 10,000 条记录批次中 99% 成功、1% 失败，全部重传产生幂等写入放大、存储写入放大和延迟放大
- **建议**: 定义预期批次大小上限，文档化故障模式影响

#### [M7] Agent 内部架构未分解

- **位置**: 第 82-96 行
- **现状**: Agent 列出了 11 项职责，但内部组件（任务管理器、缓冲管理器、上传管理器、心跳管理器）未作为正式对象
- **建议**: 补充 Agent 内部组件分解，Agent 是最复杂组件需要更细粒度的设计

#### [M8] 安全模型是叙述性而非规定性

- **位置**: 第 369-409 行
- **缺失内容**:
  - 认证协议：TLS 客户端证书？JWT？双向 TLS？
  - 令牌生命周期：如何轮换？离线三个月后是否过期？
  - 授权矩阵：各角色具体能做什么？
  - 数据传输加密终止点在哪里？
  - 静态加密：LocalBuffer 数据是否加密？
- **建议**: 安全章节需从"应该做"升级为"怎么做"

#### [M9] 多租户隔离未明确

- **位置**: 第 370 行
- **缺失内容**:
  - 租户级速率限制？
  - 存储隔离方式（逻辑隔离 vs 物理隔离）？
  - 令牌泄露后的爆炸半径？
- **建议**: 明确 Customer 级隔离的技术实现方式

### 完全缺失的领域

以下主题在文档中完全未提及：

| 领域 | 说明 |
|------|------|
| API 版本控制 | Agent v1 与 Gateway v2 通信时的兼容策略 |
| 限速与配额 | Agent 和 Customer 层面均无限速设计 |
| 数据 Schema 演进 | ConnectorType 更改输出格式后下游如何处理 |
| 灾备恢复 | Gateway 数据中心丢失时的恢复计划 |
| 测试策略 | 混沌工程、故障注入、可靠性验证方法 |
| 可观测性集成 | 指标导出格式（Prometheus/OpenTelemetry） |
| 容量规划指南 | 硬件选型和资源估算指导 |
| 数据保留与合规 | 采集数据保留期限、数据主权考虑 |
| Agent 自更新机制 | Agent 二进制文件补丁管理 |
| 配置漂移检测 | Agent 本地配置文件被手动修改的检测 |

## 文档优点

1. 核心对象模型（Customer → Agent → CollectorTask → ConnectorInstance）边界清晰
2. 三条核心链路（控制、数据、监控）定义明确
3. 数据可靠性九步处理流程逻辑健全
4. Checkpoint 与 messageId 职责分离的架构决策正确
5. 背压水位分级策略实用
6. 动态分发预留字段设计合理，v1 边界清晰
7. 异常恢复场景覆盖面较广（Agent 离线、Gateway 不可用、数据存储失败、ConnectorInstance 崩溃、Agent 重启、缓冲损坏）
